## Background — deploy vs runtime bytecode

In a real Solidity contract:

1. **Deployment bytecode** runs once when the contract is created (the
   `CREATE`/`CREATE2` payload). The constructor may do work, and at the
   end it emits the **runtime bytecode** via `RETURN` — this is what
   gets stored as the contract's code on-chain.
2. **Runtime bytecode** is the final deployed code, executed on every
   subsequent call.

Our `EvmSmith.Add3.bytecode` is runtime bytecode. To make it
callable in Foundry we'd normally need deployment bytecode that
copies the runtime into memory and `RETURN`s it. The standard stub is

```
PUSH1 <runtime-size>  PUSH1 <stub-size>  PUSH1 0x00  CODECOPY
PUSH1 <runtime-size>  PUSH1 0x00  RETURN
```

then the runtime appended. But we don't need to do this — Foundry
provides a cleaner mechanism for tests (next section).

## Deployment strategy: `vm.etch`

`vm.etch(address addr, bytes code)` is a Foundry cheat code that
installs arbitrary bytecode at `addr` with no deployment step. This is
the canonical way to test raw EVM bytecode in Foundry — Foundry's own
docs use this for "test this runtime bytecode" scenarios. It matches
our use case perfectly: we *are* the runtime bytecode and we don't
care about constructor semantics for this task.

Test setup:

```solidity
address constant ADD3 = address(0x00000000000000000000000000000000000aDD3);
function setUp() public { vm.etch(ADD3, RUNTIME); }
```

### Caveats (documented; none problematic here)

- **Nonce unchanged.** `vm.etch` installs code; it doesn't bump nonce.
  The `aDD3` address will have nonce 0. Irrelevant for a contract
  that only `RETURN`s; flagged for completeness.
- **EXTCODEHASH / EXTCODESIZE.** Recent `forge` (1.x) handles these
  correctly on etched addresses. Pin a minimum forge version
  (≥ 1.0.0) in the README to be safe.
- **No `CODECOPY`-of-self.** Verified by inspection of the 19-byte
  program (`0x60003560203560010160403560010160005260206000f3` — only
  `PUSH1`, `CALLDATALOAD`, `ADD`, `MSTORE`, `RETURN`; no `CODECOPY`
  opcode 0x39 or `CODESIZE` 0x38). If a future program uses them,
  `vm.etch` interacts cleanly with `CODECOPY` since forge makes the
  installed code the account's code as if deployed normally.

### Alternatives considered and rejected

- **Write a deployment stub in Lean + deploy via `new`**. Correct but
  adds a second bytecode to maintain and another proof surface. Out
  of scope ("we don't need to make proofs for constructors").
- **Solidity wrapper contract** with the runtime as a `bytes` constant
  and `delegatecall` into a deployed copy. Extra indirection, no
  benefit over `vm.etch`.
- **Use `CREATE` opcode inline** to deploy the runtime with a
  hand-written stub. Matches "real deployment" more closely but
  introduces the stub, its correctness, and address-computation
  subtleties.

## Bytecode sourcing — Lean → Solidity (file-based, single source of truth)

The Lean source-of-truth is
`EvmSmith/Demos/Add3/Program.lean :: bytecode` (19 bytes:
`0x60003560203560010160403560010160005260206000f3`).

### Chosen design

Add a tiny Lean exe that writes the bytecode as hex into a known file
inside the Foundry project, and have Solidity read it at `setUp` time.
Single source of truth, no drift possible.

**Lean side — `EvmSmith/Demos/Add3/DumpBytecode.lean`**:

Use the existing `EvmYul.toHex : ByteArray → String`
(`EVMYulLean/EvmYul/Wheels.lean:47`) — no `0x` prefix, no trailing
newline, so we add both:

```lean
import EvmYul.Wheels
import EvmSmith.Demos.Add3.Program

open EvmSmith

def main : IO Unit :=
  IO.FS.writeFile
    "EvmSmith/Demos/Add3/foundry/test/Add3.bytecode.hex"
    ("0x" ++ EvmYul.toHex Add3.bytecode ++ "\n")
```

Wire it up as a Lake exe target:

```lean
-- in lakefile.lean
lean_exe «add3-dump-bytecode» where
  root := `EvmSmith.Demos.Add3.DumpBytecode
```

**Solidity side — `test/Add3.t.sol`**:

```solidity
function setUp() public {
    string memory hexStr = vm.readFile("test/Add3.bytecode.hex");
    // Strip trailing newline if present, then parseBytes.
    bytes memory runtime = vm.parseBytes(_trim(hexStr));
    vm.etch(ADD3, runtime);
}
```

`vm.parseBytes` accepts the `0x`-prefixed hex string and returns raw
bytes. `vm.readFile` resolves paths relative to the project root
(`EvmSmith/Demos/Add3/foundry/`), which by default is allowed
(`fs_permissions` in `foundry.toml` is not needed for reads inside
the project, but we'll add an explicit permission line to be clean —
see `foundry.toml` section below).

**Workflow.** Commit the generated `Add3.bytecode.hex` to git so that
someone who clones the repo can `forge test` without first running
`lake exe`. Regenerate it whenever `Program.lean` changes:

```bash
lake exe add3-dump-bytecode
```

Add to `AGENTS.md`: "If you change `Add3.bytecode` in Lean, also run
`lake exe add3-dump-bytecode` to refresh the hex file."

Add a `#guard` in `Tests/Guards.lean` that, combined with the
generator, means drift surfaces as either (a) a failing guard in Lean,
or (b) a failing `forge test` because the hex file is stale:

```lean
#guard EvmSmith.Add3.bytecode.size == 19
```

(This is a cheap structural check; the generator is the actual sync
mechanism.)

### Alternatives considered and rejected

- **A. Hard-code hex in `Add3.t.sol`** with a `#guard` in Lean. The
  `#guard` can only compare the Lean-side `bytecode` to another Lean
  literal — it can't see the Solidity file, so it doesn't catch real
  drift between the two. Vacuous.
- **B. Pre-commit hook** that diffs the hex-as-extracted-from-Lean
  against the hex-in-`.t.sol`. Works but adds repo infrastructure
  (git hooks) and can be silently bypassed with `--no-verify`.
- **C. Foundry post-compile script** invoking `lake exe` during
  `forge test`. Couples the two toolchains and complicates CI.

## Test design

### Concrete tests
```solidity
function test_Add3_concrete() public {
    assertEq(_callAdd3(1, 2, 3), 6);
    assertEq(_callAdd3(10, 20, 30), 60);
    assertEq(_callAdd3(0, 0, 0), 0);
    assertEq(_callAdd3(type(uint256).max, 1, 0), 0);    // wraps
    assertEq(_callAdd3(type(uint256).max, type(uint256).max, type(uint256).max),
             type(uint256).max - 2);                     // 3·(2^256-1) mod 2^256
}
```

### Short-calldata tests (new per review)

`CALLDATALOAD` zero-pads past end-of-calldata, so the contract
tolerates short or empty input. The `compute_correct` theorem abstracts
over this via its `h0`/`h1`/`h2` hypotheses; these tests exercise that
concretely.

```solidity
function test_Add3_shortCalldata() public {
    // Empty calldata → all three reads return 0.
    (bool ok, bytes memory ret) = ADD3.call("");
    require(ok, "empty calldata should not revert");
    assertEq(abi.decode(ret, (uint256)), 0);

    // 32 bytes → b and c read zero-padded = 0.
    ret = _callRaw(abi.encodePacked(uint256(5)));
    assertEq(abi.decode(ret, (uint256)), 5);

    // 64 bytes → c reads zero-padded = 0.
    ret = _callRaw(abi.encodePacked(uint256(5), uint256(7)));
    assertEq(abi.decode(ret, (uint256)), 12);
}

function test_Add3_longCalldata() public {
    // 128 bytes: fourth word is ignored by the contract.
    bytes memory cd = abi.encodePacked(
        uint256(1), uint256(2), uint256(3), uint256(999));
    (bool ok, bytes memory ret) = ADD3.call(cd);
    require(ok);
    assertEq(abi.decode(ret, (uint256)), 6);
}

function _callRaw(bytes memory cd) internal returns (bytes memory ret) {
    bool ok;
    (ok, ret) = ADD3.call(cd);
    require(ok, "call failed");
}
```

### Fuzz test
```solidity
function testFuzz_Add3(uint256 a, uint256 b, uint256 c) public {
    unchecked {
        uint256 expected = a + b + c;   // wraps, matching EVM ADD
        assertEq(_callAdd3(a, b, c), expected);
    }
}
```

Notes:
- The contract uses raw `ADD` which wraps mod 2^256. So the expected
  value must also wrap → `unchecked { ... }` in Solidity.
- Default Foundry fuzz runs 256 iterations — plenty for three-uint
  fuzzing.

### Helper
```solidity
function _callAdd3(uint256 a, uint256 b, uint256 c) internal returns (uint256) {
    bytes memory cd = abi.encodePacked(a, b, c);    // 96 bytes, no selector
    (bool ok, bytes memory ret) = ADD3.call(cd);
    require(ok, "add3 call failed");
    assertEq(ret.length, 32);
    return abi.decode(ret, (uint256));
}
```

`abi.encodePacked(a, b, c)` produces 96 raw bytes (three 32-byte
words concatenated) — exactly what `CALLDATALOAD 0/32/64` will read.
No function selector, no ABI encoding overhead.

## Full test file (draft)

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.20;    // pinned, matches foundry.toml

import "forge-std/Test.sol";

/// @title Tests for the `add3` runtime bytecode
/// @notice The runtime bytecode is loaded from `test/Add3.bytecode.hex`,
///         which is generated by `lake exe add3-dump-bytecode` from
///         `../Program.lean :: bytecode`. The arithmetic correctness
///         of the compute prefix is proved in `../Proofs.lean`.
contract Add3Test is Test {
    address constant ADD3 =
        address(0x00000000000000000000000000000000000aDD3);

    function setUp() public {
        bytes memory runtime = vm.parseBytes(_trim(vm.readFile("test/Add3.bytecode.hex")));
        vm.etch(ADD3, runtime);
    }

    // ---- helpers ----

    function _callAdd3(uint256 a, uint256 b, uint256 c) internal returns (uint256) {
        bytes memory cd = abi.encodePacked(a, b, c);
        (bool ok, bytes memory ret) = ADD3.call(cd);
        require(ok, "add3 call failed");
        assertEq(ret.length, 32, "return size");
        return abi.decode(ret, (uint256));
    }

    function _callRaw(bytes memory cd) internal returns (bytes memory ret) {
        bool ok;
        (ok, ret) = ADD3.call(cd);
        require(ok, "call failed");
    }

    function _trim(string memory s) internal pure returns (string memory) {
        bytes memory b = bytes(s);
        uint256 n = b.length;
        // Strip trailing \n, \r (Windows CRLF), \t, space.
        while (n > 0 && (b[n - 1] == 0x0a || b[n - 1] == 0x0d
                      || b[n - 1] == 0x09 || b[n - 1] == 0x20)) { n--; }
        bytes memory r = new bytes(n);
        for (uint256 i = 0; i < n; i++) r[i] = b[i];
        return string(r);
    }

    // ---- concrete tests ----

    function test_Add3_concrete() public {
        assertEq(_callAdd3(1, 2, 3), 6);
        assertEq(_callAdd3(10, 20, 30), 60);
        assertEq(_callAdd3(0, 0, 0), 0);
    }

    function test_Add3_wraps() public {
        assertEq(_callAdd3(type(uint256).max, 1, 0), 0);
        unchecked {
            uint256 expected = type(uint256).max + type(uint256).max + type(uint256).max;
            assertEq(_callAdd3(type(uint256).max, type(uint256).max, type(uint256).max),
                     expected);
        }
    }

    function test_Add3_shortCalldata() public {
        (bool ok, bytes memory ret) = ADD3.call("");
        require(ok, "empty calldata should not revert");
        assertEq(abi.decode(ret, (uint256)), 0);

        ret = _callRaw(abi.encodePacked(uint256(5)));
        assertEq(abi.decode(ret, (uint256)), 5);

        ret = _callRaw(abi.encodePacked(uint256(5), uint256(7)));
        assertEq(abi.decode(ret, (uint256)), 12);
    }

    function test_Add3_longCalldata() public {
        bytes memory cd = abi.encodePacked(
            uint256(1), uint256(2), uint256(3), uint256(999));
        (bool ok, bytes memory ret) = ADD3.call(cd);
        require(ok);
        assertEq(abi.decode(ret, (uint256)), 6);
    }

    // ---- fuzz ----

    function testFuzz_Add3(uint256 a, uint256 b, uint256 c) public {
        unchecked {
            uint256 expected = a + b + c;
            assertEq(_callAdd3(a, b, c), expected);
        }
    }
}
```

## `foundry.toml`

We don't have a `src/` directory (no Solidity production code),
so omit `src` and let Foundry use its default. Also add explicit
`fs_permissions` for the bytecode-hex read.

```toml
[profile.default]
out = "out"
libs = ["lib"]
solc_version = "0.8.20"
optimizer = true
optimizer_runs = 200
fs_permissions = [{ access = "read", path = "./test/Add3.bytecode.hex" }]
```

If `src` is unset, `forge` uses `src/` by default but won't error on
its absence unless there's a file actually referencing it. If `forge
build` complains anyway, keep `src/` as an empty dir with a
`.gitkeep`.

## Sync check in Lean

The generator (`lake exe add3-dump-bytecode`) is the real sync
mechanism — whoever changes `Add3.bytecode` is told in `AGENTS.md` to
re-run it, and the Foundry side reads directly from the generated
file. Belt-and-suspenders: add a structural `#guard` in
`EvmSmith/Tests/Guards.lean` that catches the most likely
accidental-edit failure (bytecode size changing without thought):

```lean
#guard EvmSmith.Add3.bytecode.size == 19
```

This is cheap and not a tautology — it fires at `lake build` time if
anyone edits `Program.lean` to change the bytecode length without
updating the guard, which is the cue to also re-run the dump.

## Submodule story (corrected per review)

`forge init` pulls `forge-std` as a git submodule at
`EvmSmith/Demos/Add3/foundry/lib/forge-std`. Because we're inside an
existing git repo, `git submodule add` registers the submodule in the
**repo root's `.gitmodules`** (not in a Foundry-local one), with path
`EvmSmith/Demos/Add3/foundry/lib/forge-std`. This is standard and
acceptable.

README / AGENTS.md notes needed: cloning the repo now requires
`git clone --recursive` or a follow-up `git submodule update --init
--recursive`.

Already-gitignored `EVMYulLean/` (a manual clone) is unaffected; the
two conventions coexist.

## Risks / edge cases

- **forge-std submodule fetch**: depends on network. First `forge init`
  clones forge-std from GitHub. Works on this machine; if offline,
  the task blocks.
- **solc 0.8.20 availability**: forge auto-downloads it to
  `~/.svm/0.8.20/` on first build. Should work; may slow first run.
- **Minimum forge version**: `--no-commit` and `vm.parseBytes`/
  `vm.readFile` require ≥ 1.0.0 forge. This machine has 1.5.1; pin
  ≥ 1.0.0 in the README.
- **Fuzz counterexamples**: unlikely — the contract matches EVM
  semantics which Solidity `unchecked` also matches. If a
  counterexample does surface, it's a real finding.
- **Return data size**: the contract always returns exactly 32 bytes
  (`MSTORE` + `RETURN 0x20 0x00`). The `assertEq(ret.length, 32)`
  check catches any surprise.
- **Gas**: the contract is ~50 gas; default test gas limit is fine.
- **`fs_permissions` path resolution**: the read path in
  `foundry.toml` is relative to the Foundry project root (not the
  enclosing repo). Double-check this on first run.

## Out of scope

- Proving the deployment-bytecode version (we'd need to Lean-model the
  CODECOPY stub).
- Real `CREATE2`-based deployment inside the test. `vm.etch` is
  strictly better for this use case.
- Integration with a solidity-side ABI that would route through a
  function selector — the contract takes raw calldata by design.
- Multi-contract tests, cross-contract calls, state tests.
